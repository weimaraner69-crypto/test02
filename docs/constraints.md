# 制約仕様（Constraints）

## 目的

プロジェクトの制約（安全性、品質、運用上の制限）を定義する。本ファイルは制約の仕様正本であり、実装は本ファイルに追従する。

## 制約の適用範囲

- すべての処理は、実行前に制約を評価する。
- 判断不能な場合は安全側に倒す（フェイルクローズ）。
- 制約判定は一箇所に集約し、回避できない構造にする。

## 制約一覧

> しきい値はプロジェクト固有に設定する。初期値は ADR で根拠を明記する（ADR 未策定の場合は将来追記）。

### C-001 学年制限

| パラメータ | 値 | 備考 |
| --- | --- | --- |
| しきい値 | 1〜6（整数） | 小学校学年数 |

- 判定：`grade` が 1 未満または 6 超過の場合に発動
- 理由コード：`C001_invalid_grade`

> **備考**: `src/domain/learning.py` の `validate_grade()` にて実装済み（grade が 1〜6 範囲外の場合に `ValidationError` を送出）。`LearningService.get_content_for_grade()` は学年制約とは別のロジックであり、本制約の適用範囲外。

#### C-001 閾値定義

| 閾値 | トリガー条件 | アクション |
| --- | --- | --- |
| grade < 1 | grade が 1 未満 | `ValidationError` を送出して処理を中断 |
| grade > 6 | grade が 6 超過 | `ValidationError` を送出して処理を中断 |

> **注記**: `ValidationError` 送出時は `reason_code` 属性に `"C001_invalid_grade"` を設定すること。N-024 にて `ValidationError.__init__(message, reason_code=None)` を `src/core/exceptions.py` に実装済み（FR-030/031 の `reason_code` と共通）。`domain/learning.py` の `validate_grade()` への `reason_code="C001_invalid_grade"` 付与は Backlog に別途追加予定。

### C-002 コンテンツ安全性

| パラメータ | 値 | 備考 |
| --- | --- | --- |
| しきい値 | 不適切表現フラグ数 0 件 | 子供向けアプリのため不適切コンテンツを排除 |

- 判定：Gemini API レスポンスに不適切表現フラグが含まれる場合に発動
- 理由コード：`C002_unsafe_content`

> **注記**: N-023（PR #43）にて実装済み。`finish_reason` が `SAFETY` または `FinishReason.SAFETY` の場合に `ValidationError(reason_code="C002_unsafe_content")` を送出。

#### C-002 閾値定義

| 閾値 | トリガー条件 | アクション |
| --- | --- | --- |
| 不適切表現フラグ > 0 | Gemini API レスポンスに不適切表現フラグが含まれる | レスポンスを破棄し `ValidationError` を送出 |
| finish_reason=SAFETY | Gemini API が `finish_reason=SAFETY` を返した | レスポンスを破棄し `ValidationError` を送出 |

### C-003 API レート制限

| パラメータ | 値 | 備考 |
| --- | --- | --- |
| しきい値（ユーザー単位） | 10 req/min | Gemini API の Free Tier 上限（60 req/min）の 1/6 をユーザーあたり上限に設定 |
| しきい値（全体） | 50 req/min | Free Tier 上限 60 req/min に対する安全マージン（同時利用 5 ユーザー相当） |

- 判定①：1 ユーザーが 60 秒以内に 10 回を超えて Gemini API を呼び出した場合に発動
- 判定②：全ユーザー合計の Gemini API 呼び出しが 60 秒以内に 50 回を超えた場合に発動
- 理由コード：`C003_rate_limit_exceeded`

> **注記**: 将来実装予定。N-026（Flask セッション管理）はログインセッションが主題でレート追跡を含まない。レート追跡は Backlog に別途追加予定。

#### C-003 閾値定義

| 閾値 | トリガー条件 | アクション |
| --- | --- | --- |
| ユーザー req/min > 10 | 1 ユーザーの Gemini API 呼び出しが 60 秒以内に 10 回超過 | 新規リクエストを拒否し `RateLimitError`（`DomainError` のサブクラス）を送出 |
| 全体 req/min > 50 | 全ユーザー合計の Gemini API 呼び出しが 60 秒以内に 50 回超過 | 新規リクエストを拒否し `RateLimitError` を送出 |
| ユーザーへのメッセージ | 上記トリガー時 | 「しばらく時間をおいてから再試行してください」メッセージを含める |

### C-004 セッション時間制限

| パラメータ | 値 | 備考 |
| --- | --- | --- |
| 警告しきい値 | 60 分（3600 秒） | 子供の集中力と疲労を考慮した警告タイミング |
| 強制終了しきい値 | 120 分（7200 秒） | 長時間利用を防ぐための強制終了タイミング |

- 判定①：継続時間が 60 分を超えた場合に警告を発動（セッションは継続）
- 判定②：継続時間が 120 分を超えた場合に強制終了を発動
- 理由コード：`C004_session_timeout`

> **注記**: 将来実装予定。N-026（Flask セッション管理）はログインセッションの維持と Cookie/SECRET_KEY 設定が主題であり、経過時間監視を含まない。セッション時間制限は Backlog に別途追加予定。

#### C-004 閾値定義

| 閾値 | トリガー条件 | アクション |
| --- | --- | --- |
| > 3600 秒（警告） | セッション継続時間が 60 分超過 | 警告メッセージをユーザーに表示し、新規問題生成を一時停止（セッション継続） |
| > 7200 秒（強制終了） | セッション継続時間が 120 分超過 | セッションを強制終了し、進捗を保存してログアウト |

### 記入例: C-099 日次損失制限

| パラメータ | 値 | 備考 |
| ---------- | --- | --- |
| しきい値 | -3% | ADR-0005 にて根拠を記載 |

- 判定：当日の累計損失率がしきい値以下になった場合に発動
- 理由コード：`C099_daily_loss_limit`

#### C-099 閾値定義

| 閾値 | トリガー条件 | アクション |
| --- | --- | --- |
| -2%（警告） | 当日の累計損失率が -2% を下回った | ログに警告を記録し、管理者に通知する |
| -3%（停止） | 当日の累計損失率が -3% を下回った | 新規操作を停止し、状態を DEGRADED に遷移する |
| -5%（緊急） | 当日の累計損失率が -5% を下回った | 全操作を停止し、状態を HALTED に遷移する |

> **注意**: 上記は記入例です。プロジェクトに適用する際は、具体的なドメインの制約に置き換えてください。

## 状態遷移（必要な場合）

```text
RUNNING ──(品質低下)──→ DEGRADED ──(制約違反/異常連鎖)──→ HALTED
   ↑                      │                               │
   └────(手動復帰)────────┘                               │
   └──────────────(手動復帰)───────────────────────────────┘
```

| 状態     | 説明     | 新規操作             |
| -------- | -------- | -------------------- |
| RUNNING  | 通常稼働 | 許可（制約評価あり） |
| DEGRADED | 品質低下 | 制限                 |
| HALTED   | 安全停止 | 禁止                 |

## 実装要件

- 制約判定は一箇所に集約し、ドメインロジックが回避できない構造にする
- 判定結果には拒否理由コードを付与する（監査ログ用）
- 制約は境界値テストを必須とする（しきい値ちょうど、直上、直下）
- 状態遷移のテストを必須とする（該当する場合）

## 見直し

- しきい値は検証結果に基づき更新する
- 更新時は plan と ADR の整合を取る
