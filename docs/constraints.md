# 制約仕様（Constraints）

## 目的

プロジェクトの制約（安全性、品質、運用上の制限）を定義する。本ファイルは制約の仕様正本であり、実装は本ファイルに追従する。

## 制約の適用範囲

- すべての処理は、実行前に制約を評価する。
- 判断不能な場合は安全側に倒す（フェイルクローズ）。
- 制約判定は一箇所に集約し、回避できない構造にする。

### 記入例: C-099 日次損失制限

| パラメータ | 値 | 備考 |
| ---------- | --- | --- |
| しきい値 | -3% | ADR-0005 にて根拠を記載 |

- 判定：当日の累計損失率がしきい値以下になった場合に発動
- 理由コード：`C099_daily_loss_limit`

#### C-099 閾値定義

| 閾値 | トリガー条件 | アクション |
| --- | --- | --- |
| -2%（警告） | 当日の累計損失率が -2% を下回った | ログに警告を記録し、管理者に通知する |
| -3%（停止） | 当日の累計損失率が -3% を下回った | 新規操作を停止し、状態を DEGRADED に遷移する |
| -5%（緊急） | 当日の累計損失率が -5% を下回った | 全操作を停止し、状態を HALTED に遷移する |

> **注意**: 上記は記入例です。プロジェクトに適用する際は、具体的なドメインの制約に置き換えてください。

## 状態遷移（必要な場合）

```
RUNNING ──(品質低下)──→ DEGRADED ──(制約違反/異常連鎖)──→ HALTED
   ↑                      │                               │
   └────(手動復帰)────────┘                               │
   └──────────────(手動復帰)───────────────────────────────┘
```

| 状態     | 説明     | 新規操作             |
| -------- | -------- | -------------------- |
| RUNNING  | 通常稼働 | 許可（制約評価あり） |
| DEGRADED | 品質低下 | 制限                 |
| HALTED   | 安全停止 | 禁止                 |

## 実装要件

- 制約判定は一箇所に集約し、ドメインロジックが回避できない構造にする
- 判定結果には拒否理由コードを付与する（監査ログ用）
- 制約は境界値テストを必須とする（しきい値ちょうど、直上、直下）
- 状態遷移のテストを必須とする（該当する場合）

## 見直し

- しきい値は検証結果に基づき更新する
- 更新時は plan と ADR の整合を取る
