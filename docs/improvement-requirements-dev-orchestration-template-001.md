## 0. 要約（結論）

本リポジトリは、**設計思想（SSOT / HITL / 監査分離 / Bounded Recursion）** が非常に優れており、土台はすでに高水準です。
一方で、★5未達の主要因は「設計の強さに対して実行系が薄い」点に集約されます。

- CI品質ゲートはテンプレート定義の初期状態として提供されており、「実プロジェクト適用後にどのように有効化・必須化するか」の指針がリポジトリ内で十分に明文化されていない
- セキュリティ自動検証は secret 検証のみ稼働中（`ci/policy_check.py` L61-66）で、SCA/SBOM/署名検証は未実装
- エージェント間通信が非構造（自然言語中心）で再現性が弱い
- 可観測性がアプリ中心で、**エージェント・パイプライン自体**の計測が不足

したがって、★5化の最短ルートは次の3本柱です。

1. **実行保証**: CI/CDを「必ず落ちる/必ず止まる」品質ゲートへ完全移行
2. **構造化**: エージェント通信と状態遷移を型安全・検証可能へ移行
3. **証跡化**: 監査・供給網・運用メトリクスを永続化し、再現可能性を担保

## 2. 最新動向（2025-2026）を踏まえた改善原則

以下は、現在の実務で有効性が高い潮流を本テンプレートに適用する際の要点です。

> **ステータス凡例**: 各項目は断りのない限り **現状: 未採用 / 推奨: 採用** を意味する。現在のリポジトリに実装済みの項目には `[実装済]` を明記する。

### 2.1 AIエージェント実装の潮流

- **Structured Outputs前提**: エージェント間連携はJSON Schema/Pydanticで契約化
- **Stateful Orchestration**: 再実行可能な状態機械（途中再開/失敗回復）
- **Evals as CI Gate**: ユニットテスト同等に「プロンプト/エージェント品質」を評価
- **Model Routing**: 高価モデルは判断点のみに限定、実装/テストは高効率モデルへ

### 2.2 DevSecOps/サプライチェーンの潮流

- **SLSA / in-toto / Sigstore** を用いた provenance・署名・検証
- **SBOM必須運用**（生成するだけでなく、ポリシーで検証）
- **Policy as Code**（OPA/Conftest等）でレビュー依存を排除
- **Secret/SCA/Container/License** のマルチレイヤ自動スキャン

### 2.3 観測・運用の潮流

- **OpenTelemetry GenAI計測**（token/cost/latency/tool-call）
- **DORA + Agent KPI** を同じダッシュボードで管理
- **Budget Guardrail**（月次/PR単位予算・超過時停止）

## 3. ★★★★★到達のための処理フロー再設計（提案）

現行の「中央集約オーケストレータ」思想は維持しつつ、故障耐性と再現性を向上させる。

> **注意**: 本フローは提案（To-Be）であり、現行の `.github/agents/orchestrator.agent.md` に記述された「Step 3: implementer → Step 4: test-engineer」の逐次フローと一部異なる（特に Step 2 の TDD ゲート化）。採用する場合は ADR として記録し、既存の正本ドキュメントへの反映計画を明示すること（P-030 正本優先原則）。

### 3.1 推奨フロー（To-Be）

1. **Plan**: orchestrator が課題分解＋受入条件生成（構造化）
2. **Spec/Test先行**: test-engineer が先にテスト観点を確定（TDDゲート）
3. **Implement**: implementer が実装（途中で静的解析を逐次実行）
4. **Parallel Audit**: spec/security/reliability を並列実行
5. **Policy Gate**: Policy-as-Code + SCA + secret + SBOM + 署名検証
6. **Eval Gate**: 代表シナリオのエージェント評価（回帰比較）
7. **Release Decision**: release-manager が証跡付きで最終判定
8. **HITL**: 人間承認後にマージ/昇格
9. **Post-merge Observe**: KPI記録・週次改善ループへ投入

### 3.2 構造面の改善ポイント

- エージェント入出力を **JSON Schema** で統一し、フォーマット逸脱は即fail
- ループ回数だけでなく、**失敗理由コード**（POLICY_FAIL等）を状態に記録
- ロールバック手順を「失敗タイプ別」に定義（実装失敗/監査失敗/供給網失敗）
- 監査結果をPRコメントだけでなくアーティファクトとして保存

## 4. 観点別の具体策（全項目★5化）

## 4.1 アーキテクチャ設計（★4→★5）

- SPOF緩和: orchestratorを維持しつつ、監査/CIイベントはイベント駆動で非同期処理
- runbookに**ロールバック決定木**を追加（どの失敗で何を戻すか）
- implement/testの逐次依存を解消し、test-engineerを前段化

**完了条件**

- 失敗シナリオ3種で復旧訓練が通る
- 監査3系統が常時並列で実行される

## 4.2 AIエージェント設計（★4→★5）

- 応答を schema-validated JSON 化
- エージェント状態を永続化（step/loop/decision/evidence）
- モデルルーティング導入（高コストモデルの利用点最小化）
- 「再現実行ID」を導入し、同一条件での再走を可能にする

**完了条件**

- フォーマット不正の見逃しゼロ
- 同一入力で同一遷移ログを再現可能

## 4.3 セキュリティ（★3→★5）

- `policy_check` のスキャン対象拡張（`.github/`を含む）
- secret検出を現行トークン形式へ更新
- SCA（依存脆弱性）をPR必須化
- SBOM生成 + 署名 + 検証をリリース条件へ統合
- actions pinning をデプロイ系 workflow（`production.yml` / `staging.yml`）に適用（`ci.yml` では既に commit hash で固定済み）

**完了条件**

- すべてのPRで secret/SCA/SBOM/署名検証が自動実施
- 失敗時は保護ブランチ上でマージ不可

## 4.4 CI/CD品質（★2→★5）

- コメントアウトされた品質チェックを、`bootstrap.sh` 実行時にプロジェクトの言語・ツールに応じて自動有効化する仕組みを追加（テンプレートのコメントアウト状態は複数言語対応のための意図的設計であり、`bootstrap.sh` 実行後は言語別の完全な CI に置き換わる）
- `concurrency` と依存キャッシュを導入
- 変更影響ベースのテスト選択 + フルテスト夜間実行
- production 記録は GitHub Actions ログ（標準出力）として保存・検索可能だが、デフォルト 90 日の保存期間を超える長期保存や高度な検索が必要な場合は外部永続ストレージへのエクスポートを追加

**完了条件**

- CI失敗のままマージ不能
- 平均CI時間と再実行率が可視化され改善傾向

## 4.5 コード/テスト設計（★4→★5）

- mutation testing を導入しテスト有効性を測定
- 型検査をstrict化し曖昧型を削減
- property testの対象を主要ドメインに拡大

**完了条件**

- mutation score しきい値到達
- 型検査での例外運用を縮小

## 4.6 ドキュメント管理（★4→★5）

- `requirements` / `constraints` のテンプレート値を実値化するチェックリスト追加
- モデル名の静的一覧を縮小し、設定ファイルをSSOT化
- 変更時の「更新必須ドキュメント」ルールをCIで検証

**完了条件**

- テンプレート未埋め項目でCI fail
- 実装差分と仕様差分の乖離を自動検知

## 4.7 運用/可観測性（★2→★5）

- OTelでagent spanを収集（step latency/token/cost/error）
- Budget cap（PR/日/週）を定義し超過時停止
- 監査ログを外部保管し、検索可能にする

**完了条件**

- 1PRあたりのコストと成功率が追跡可能
- 監査イベントの欠損率ゼロ

## 4.8 最新動向整合性（★4→★5）

- Evals（回帰ベンチ）をCIへ統合
- AI Gateway層でレート制御/フォールバック/キャッシュ
- 型安全オーケストレーション（状態機械）を段階導入

**完了条件**

- エージェント品質を継続的に数値比較可能
- モデル障害時の自動退避が機能

## 7. 最終提言

本リポジトリを全項目★5へ引き上げる鍵は、**「優れた思想」を「強制力のある運用実装」に変換すること**です。
特に優先度が高いのは以下です。

1. CI品質ゲートの完全有効化（コメントアウト運用の終了）
2. セキュリティ供給網対策の自動強制（SBOM/署名/検証）
3. エージェント通信/状態の構造化（schema + state persistence）
4. OTelとコストガードレールによる運用可視化

この4点を順に実装すれば、処理フロー・構造・運用の三位一体で成熟し、全観点を実運用レベルで ★★★★★ に到達可能です。

## 8. Copilotレビュー指摘への対応（反映事項）

本改訂では、PRレビューで指摘されやすい「抽象論に寄りすぎる」「実行順序が不明」「完了判定が曖昧」を解消するため、以下の運用ルールを明示する。

### 8.1 このリポジトリ内の実体ファイルへのマッピング

| 改善テーマ               | 主要な反映先（既存ファイル）                          | 反映内容                                                 |
| ------------------------ | ----------------------------------------------------- | -------------------------------------------------------- |
| CI品質ゲート             | `.github/workflows/ci.yml`                            | lint/format/type/test を必須化し、失敗時マージ不可にする |
| 本番昇格の証跡           | `.github/workflows/production.yml`                    | 署名検証・昇格記録の永続化（artifact/外部保管）          |
| ポリシー検査強化         | `ci/policy_check.py`                                  | `.github/` スキャン追加、現行トークン形式の検出改善      |
| オーケストレーション仕様 | `docs/orchestration.md`                               | Structured output と状態遷移（失敗理由コード）を明文化   |
| 復旧手順                 | `docs/runbook.md`                                     | 失敗タイプ別ロールバック決定木を追加                     |
| 要件・制約の実値化       | `docs/requirements.md` / `docs/constraints.md`        | テンプレート値を実値に置換し、未入力をCIでfail           |
| 可観測性                 | `docs/observability-guide.md` / `src/observability/*` | Agent span・token/cost/latency を収集対象に追加          |

### 8.2 完了判定（DoD）を数値で固定

- CI/CD: 必須チェック未通過PRのマージ率 **0%**
- セキュリティ: high/critical未解消のまま本番昇格 **0件**
- エージェント: schema違反率 **0%**（失敗は即fail-fast）
- 運用: 監査イベント欠損率 **0%**、PR単位コスト可視化率 **100%**
- テスト品質: mutation score のしきい値を設定し、未達PRはfail

### 8.3 実装優先順位（依存関係付き）

1. `ci.yml` 必須ゲート化（他施策の土台）
2. `policy_check.py` 強化 + secret/SCA導入（安全性の最低ライン）
3. `production.yml` の署名検証・昇格証跡（供給網保証）
4. エージェントI/OのSchema化 + 状態永続化（再現性）
5. OTel + Budget guardrail + Evals常設（継続改善）

> これにより、改善施策が「提案」ではなく「実装順序と合格条件を持つ実行計画」として運用可能になる。
